# PandaMUX: Bug Fixes and UX Overhaul (July 2026 review spec)

## Context

Chaz and Brandon's July 2026 review produced a handoff spec ("Pandemx") covering six bug clusters and seven UX changes. The spec was written against the deleted Electron/xterm.js prototype (its implementation notes reference FitAddon, ResizeObserver, Electron clipboard wiring); this plan maps every item onto the current native Rust app (Iced + alacritty_terminal + portable-pty + russh). Nothing is deferred: all of Part 1 (1.1 half-height rendering, 1.2 scrollback, 1.3 selection/copy/paste/context menu, 1.4 cross-machine project duplication, 1.5 empty state, 1.6 new-tab latency) and Part 2 (2.1 sidebar/tab model, 2.2 launcher modal, 2.3 favorites/recents, 2.4 Home/Projects/Type rail, 2.5 Home dashboard, 2.6 keyboard shortcuts, 2.7 session-type chooser) is in scope.

Note on the "multiple machines" framing: in PandaMUX one app instance reaches other machines over SSH profiles. "Opening SupportForge from machine B" means a second checkout reached via a different transport or SSH profile from the same app, so project identity can live in app-side state; no cross-machine sync layer is needed.

## Ground truth (verified against the code, July 16 2026)

All three renderer complaints are real gaps in the Rust build, not Electron leftovers:

- **1.1 root cause**: every terminal grid is hardcoded `GridSize::new(120, 30)` (backend.rs:740,1060,1880; iced_runtime.rs:389,1710,1732; main.rs:109; ssh.rs:1212). The canvas viewport ([iced_shell.rs:284-529](crates/pandamux-ui/src/iced_shell.rs)) draws exactly snapshot rows x cols at hardcoded cell metrics (CELL_WIDTH 7.8, CELL_HEIGHT 19.0 in theme.rs) and never reads its own bounds. No window-resize or scale-factor handling exists anywhere. Resize plumbing exists but is reachable only via the pipe RPC (`surface.resize`, backend.rs:1368-1382). Two footguns: local `PtySessionManager::resize` recreates the grid and wipes screen+scrollback (pty.rs:123-140); SSH resize sends window_change but never resizes the local grid (ssh.rs:385-401).
- **1.2**: engine scrollback exists but is `rows * 4` lines (grid.rs:131), not configurable, and wiped on resize. The UI renders only `visible_cells()`: no display-offset state, no wheel handling (the canvas Program implements only `draw`), no scrollbar, no Shift+PageUp (PageUp always sends `ESC[5~`), so the view always tails.
- **1.3**: no mouse events reach the terminal canvas at all; no selection, no highlight, Ctrl+C always sends 0x03, no Ctrl+Shift+C/V, no OS-clipboard paste binding, no right-click handling or context-menu widget anywhere. Already working and reusable: bracketed paste (`wrap_paste`), OSC 52 copy to OS clipboard incl. over SSH (`drain_clipboard_stores`), the arboard bridge (clipboard_os.rs), `surface.paste`/`clipboard.*` RPCs, and alacritty_terminal's built-in Selection API (wrapped, never leaked).
- **1.4 root cause**: there is no project entity; a workspace IS the project, keyed by `ProjectKey::from_location` (project.rs:34-59), which is `local:<path>` or `ssh:<random-profile-uuid>:<path>`. Identity is "how this machine reached the folder", so the same repo via two transports or a re-imported host duplicates.
- **1.5**: core guards forbid closing the last workspace/pane/surface (state.rs:429,498,708) and the UI close cascade no-ops on the last surface (iced_runtime.rs:1324-1329). No empty state exists, by design.
- **1.6**: local launches are synchronous with no connecting feedback; SSH opens one full TCP+auth russh connection per surface (ssh.rs:189-252), no per-host reuse, durability via per-surface remote tmux. Only SSH has a Launching state.
- **2.1/2.4**: the session rail already renders one entry per surface grouped by Project/Type/Host (session_panel.rs:24-45), host is already a per-entry field; but names are derived and not renameable, and there is no Home view.
- **2.2/2.7**: the recently added launcher (session_launcher.rs) covers connection + folder steps only; no type step (local always spawns PowerShell), no favorites/recents, no arrow-key navigation. Claude/Codex/Gemini exist only as unpersisted `agent.spawn` command strings; there is no session-type concept.
- **2.5**: not expressible today; surfaces are owned by value in exactly one workspace's split tree (split_tree.rs:29-33), no by-ID references. But per-surface snapshots are already a map keyed by surface id, so live rendering of any surface from a second layout is feasible.
- **2.6**: shortcuts are hardcoded match arms (`shortcut_for`, iced_runtime.rs:2046-2063) plus a separate hand-maintained label list in settings.rs:68-85 that has already drifted. No cheat sheet, no Ctrl+1..9/Ctrl+0/Ctrl+Tab, Ctrl+W closes without confirm. No persistent settings file of any kind exists (theme/accent are in-memory only).

Knowledge-base constraints (LL-G/BP, checked): no blocking std::fs on the tokio executor for new IO (use spawn_blocking or std threads outside async, matching existing store patterns); child processes spawned from the GUI build need CREATE_NO_WINDOW semantics (PTY spawning already handles console allocation); any new registry must have a single source of truth in AppState/pandamux-app, no parallel stores.

## Decisions on the spec's open questions

The spec authorizes resolving these during implementation and gives recommendations; adopted as follows:

1. **Project identity (1.4)**: explicit `ProjectId` as a config entry (the spec's recommended "explicit ID", in its config-entry form) held in a project registry inside `AppState`, with matching heuristics on first open (git remote URL when cheaply readable, else normalized folder leaf name) and a manual merge/split/rename affordance. No dotfile markers written into user repos.
2. **Favorites/recents (2.3)**: per machine for v1; decision noted in a code comment at the store definition.
3. **Home panes (2.5)**: true live multi-view. Rendering is snapshot-map based and only one view is visible at a time, so a Home layout that references surfaces by ID renders the live session cheaply; no navigate-on-focus fallback needed.
4. **Ctrl+number passthrough (2.6)**: reserved app-level keybindings (Windows Terminal model) with a settings-based passthrough override.

## Implementation plan

Three workstreams, sequenced in "Suggested order" at the end. Workstream A: terminal renderer/engine (1.1, 1.2, 1.3, 1.6). Workstream B: project model + UX restructure (1.4, 1.5, 2.1-2.5, 2.7). Workstream C: keymap + settings pipeline (2.6 + the settings store A and B consume).

### Workstream A: terminal renderer/engine (1.1, 1.2, 1.3, 1.6)

Version facts confirmed: alacritty_terminal `=0.26.0` provides `Term::resize` (content-preserving reflow), `scroll_display(Scroll::{Delta,PageUp,PageDown,Top,Bottom})`, built-in `Selection` + `selection_to_string()`, `display_offset`, `TermMode` flags, `set_options` for live history changes. iced `=0.14.0` canvas `Program::update` receives every event (incl. `RedrawRequested`) plus `bounds`, and can publish messages; `Paragraph::with_text`/`min_bounds` measures real glyph metrics headless. russh `client::Handle` is not Clone but `channel_open_session(&self)` works through an `Arc`. No new dependencies anywhere. Corrected location: footgun 1 is in session.rs:138, not pty.rs.

**A0. Engine foundations (pandamux-term; fixes both footguns, unblocks everything):**
- `TerminalGrid::resize(size)` calling `Term::resize` (reflow, history intact). session.rs:123-140 replaces grid recreation with it; ssh.rs:385-401 additionally resizes the local grid before sending `window_change`, and the driver updates its stored size so reconnect re-requests the PTY at the current size. SSH reset-on-reattach keeps recreating the grid (server repaint needs a clean buffer; documented limitation, tmux holds history server-side).
- Configurable scrollback: `DEFAULT_SCROLLBACK_LINES = 10_000` replaces `rows * 4`; `with_scrollback`/`set_scrollback` (via `set_options`); `set_scrollback_lines` on both managers, driven by `terminal.scrollback_lines` from the Workstream C settings store.
- Scroll API: `ScrollAmount { Lines(i32), PageUp, PageDown, Top, Bottom }`, `scroll_display`, `display_offset()`. No-yank-while-scrolled-up is native alacritty behavior.
- Extended `ScreenCells`: rows rendered AT the display offset, plus `cursor_visible`, `display_offset`, `history_size`, `selection: Vec<SelectionSpan>`, `modes: TermModes { alt_screen, mouse_reporting, alternate_scroll, app_cursor, bracketed_paste }`. All wrapper types exported from pandamux-term; alacritty types never leak.
- Selection API: `start_selection/update_selection/clear_selection/has_selection/selection_text/select_all/clear_buffer` with `SelectionMode { Simple, Word, Line }`, display coordinates converted internally; alacritty rotates selections on scroll/resize/output. Manager passthroughs on both `PtySessionManager` and `RemoteSessionManager`.
- `DEFAULT_GRID_SIZE` constant replaces the nine hardcoded `GridSize::new(120, 30)` sites.

**A1. Sizing (1.1):**
- New `crates/pandamux-ui/src/metrics.rs`: `CellMetrics::measure(font_size)` using `iced::advanced::graphics::text::Paragraph` over a 64-char mono run (logical pixels, scale-factor independent, headless); `get()` cached with a fallback to the old 7.8/19.0 constants. All draw/hit-test math routes through it.
- `TerminalViewport` becomes `canvas::Program<ShellMessage>` with `type State = ViewportState`; gains `surface_id` + `metrics`. `update` derives `cols/rows = floor((bounds - 2*padding) / cell)` (clamped to 20x5 floor) on `RedrawRequested`/mouse events and publishes `ViewportResized { surface_id, columns, rows }` only when it differs from both the snapshot dims and the last published value (idempotent against canvas state resets).
- Runtime keeps `viewport_sizes: HashMap<SurfaceId, GridSize>` + `pending_resizes` debounced ~2 ticks (~200ms, absorbs window drags), flushed on Tick to `remotes.resize` or `ptys.resize`. Spawn paths (`sync_terminal_sessions`, `project_launcher::launch_local/launch_remote_blocking` gain a `size` param) use the last known viewport size instead of 120x30.
- `draw` swaps hardcoded cell constants for `CellMetrics`. This alone plus A0 makes the pane fill correctly and keeps PTY dims in sync on window resize, split/unsplit, sidebar toggle, and zoom (all flow through canvas bounds; no window-resize subscription needed).

**A2. Scrollback UX (1.2):**
- `TerminalSnapshot` carries `display_offset/history_size/selection/cursor_visible/modes`.
- Messages: `ViewportScrolled { surface_id, lines }` (wheel/trackpad; fractional accumulator in ViewportState, pixels divided by cell height), `ViewportScrollTo { offset }` (scrollbar drag, jump pill), `ScrollPageFocused(-1/1)` (Shift+PageUp/PageDown in `map_named_key`; unshifted keeps sending `ESC[5~/6~`), `ScrollToBottomFocused`.
- Alt-screen handling: history is 0 on the alt screen so the scrollbar/pill self-hide; wheel translates to 3x arrow keys when `alt_screen && alternate_scroll && !mouse_reporting` (DECCKM-aware), matching Windows Terminal.
- Typing snaps to bottom: `scroll_display(Bottom)` at the top of `write_terminal_input`.
- Scrollbar drawn inside the canvas (6px track at right edge, min 24px thumb, visible on hover/recent-scroll/drag, fade via `request_redraw_at`); draggable thumb + track paging via ViewportState drag tracking; unit-testable pure geometry helper. Jump-to-bottom pill bottom-right when `display_offset > 0`.
- Scroll position and buffer survive tab switches for free (sessions keyed by surface id stay alive; offset is engine state).

**A3. Selection, clipboard, context menu (1.3):**
- Canvas `update` hit-tests cells (`cell_at(point, bounds, metrics) -> (line, col, right_half)`, unit-testable): left press starts Simple/Word/Line selection by click count (400ms/4px multi-click detection in ViewportState), drag publishes `SelectionUpdated` deduped at cell granularity, drag past the edge auto-scrolls into scrollback, release finishes. `mouse_reporting && !shift` suppresses local selection (Shift forces it, Windows Terminal behavior). Selection spans painted after cell backgrounds, before find highlight and text.
- Shortcuts added to `shortcut_for` (which already beats `control_byte`): Ctrl+C -> `CopyOrInterrupt` (copy+clear-selection if `has_selection`, else 0x03), Ctrl+Shift+C -> copy only, Ctrl+V and Ctrl+Shift+V -> `PasteRequested`. Paste: `clipboard_os::get_text()`, newline normalization to `\r` ONLY when bracketed paste is inactive, `wrap_paste` otherwise, snap to bottom, chunked writes (local already chunks; remote payload chunked <= 16KiB with bracket markers at the ends); generous size cap with a notification. Works identically in Claude Code panes (plain PTY surfaces with bracketed paste).
- Context menu: new `crates/pandamux-ui/src/context_menu.rs` + `ContextMenuRequested { surface_id, x, y }` from canvas right-press (never writes to the PTY, never auto-pastes; no auto-paste path exists anywhere). Full-window backdrop mouse_area for dismiss + absolutely positioned card (same stack-layer technique as the notifications panel), items: Copy (disabled without selection), Paste, Select All, Clear Buffer, Find, Split Right, Split Down, Close Tab, all dispatching to existing handlers. Deliberately NOT a `chrome::Overlay` (that enum gates typing); Esc and typing close it first.
- Clear Buffer = engine `clear_buffer` + a form feed to the PTY so the prompt repaints. Copy-mode stub stays untouched (mouse selection supersedes it; keyboard copy-mode is a natural follow-up on the new API).

**A4. Latency (1.6):**
- Instrumentation first (produces before/after numbers): `LaunchTimeline` marks intent/spawned/first_output/interactive per surface, one eprintln line per launch under `PANDAMUX_TIMING=1`.
- Local connecting feedback: pane exists within one frame already; the gap is a blank pane until the prompt. `terminal_snapshots` synthesizes "Starting {shell}..." for young blank sessions and "Connecting to {host}..." / "Reconnecting..." for `RemoteStatus::Connecting/Retrying`, giving sub-100ms visible feedback for local and SSH.
- SSH per-host connection reuse: `HostPool { HashMap<HostKey, Arc<client::Handle>> }` inside `RemoteSessionManager` (HostKey = host/port/user/auth-tag, never the password; single-flight dial under the lock; `is_closed()` liveness; invalidate on closed handle). Per-surface drivers stop dialing and just `channel_open_session` + `request_pty` + tmux exec, so a second session on a connected host skips TCP+KEX+auth entirely. Kill path stops calling `handle.disconnect` (shared now): channel eof+close instead; v1 leaves idle pooled connections open (keepalive 15s, no inactivity timeout). SFTP and the launcher's folder browsing route through the pool too, which pre-warms the connection during folder picking. Reconnect keeps per-driver backoff; single-flight prevents thundering herds. `connect_ready` timeout must not tear down the shared handle.

Workstream A risks: canvas widget state dies on tree shape changes (in-flight drags die, acceptable; resize publishing is idempotent); mouse-mode SGR click forwarding to TUI apps is scoped as a fast-follow (v1: wheel-as-arrows on alt screen, Shift forces selection); IME composed text unaffected (only ctrl+c/v intercepted); re-measure CellMetrics if a font-size setting lands later.

### Workstream B: project model + UX restructure (1.4, 1.5, 2.1-2.5, 2.7)

### Workstream C: data-driven keymap, cheat sheet, settings pipeline (2.6)

**C1. Core keymap module (new `crates/pandamux-core/src/keymap.rs`).** Framework-agnostic types consumed by BOTH the event decoder (pandamux-app) and every display surface (pandamux-ui), so drift is impossible by construction: `Mods`, `KeySpec { Char(char), Digit(u8), Named(NamedKey) }` (Digit matches the PHYSICAL digit-row key via iced 0.14's `physical_key`, so Ctrl+1..9 works on AZERTY), `KeyChord` with `Display`/`FromStr` ("ctrl+shift+c", "ctrl+1", "shift+pageup") and string serde, `Action` enum with stable string ids/labels/categories and an `allowed_with_overlay()` allowlist, `Keymap { bindings, pass_through }` with `defaults()`, `with_settings()` (user overrides; bad entries warn and keep defaults), `resolve(KeyInput)`, `chord_for/display_for/sections()`. Default table = all 12 existing bindings plus: Ctrl+Enter ZoomPane, Ctrl+Tab/Ctrl+Shift+Tab next/prev tab, Ctrl+1..9 FocusProject(n), Ctrl+0 and Ctrl+Home GoHome, Ctrl+/ and F1 cheat sheet, Ctrl+C CopyOrInterrupt, Ctrl+Shift+C Copy, Ctrl+V and Ctrl+Shift+V Paste, Shift+PageUp/PageDown scroll. Unit test asserts unique chords in the default table.

**C2. Core settings + SettingsStore.** `UserSettings { version, ui: UiSettings, terminal: TerminalSettings, keyboard: KeyboardSettings }` in new `crates/pandamux-core/src/settings.rs`, every field `#[serde(default)]`: `ui.theme/accent/showStatusBar` (strings, mapped to the iced-side enums in pandamux-app), `terminal.scrollbackLines` (10000, clamped 1k..200k), `terminal.welcomePromptEnabled` (true), `terminal.rightClickPasteOptin` (false; menu is the default, auto-paste stays off unless opted in), `terminal.confirmCloseOnRunning` (true), `keyboard.passThrough: Vec<String>` + `keyboard.overrides: BTreeMap<String, Option<String>>` (action id -> chord or null to unbind). Dotted-path `settings_get/settings_set` helpers in core (serde_json walk) shared by RPC and tooling. `SettingsStore` -> `config/settings.json` mirroring `SshProfileStore` (versioned, .bak migration, atomic write, corrupt file preserved + saves refused). Load once synchronously in `NativeShellRuntime::new` (before iced starts); saves via `Task::perform` + `spawn_blocking` (honors the no-blocking-fs rule), debounced ~500ms on the existing Tick, flushed on window close.

**C3. Runtime key refactor.** `map_key_event` is a stateless fn pointer, so a rebindable keymap cannot live there: it now only converts the iced event into `ShellMessage::KeyPressed(KeyInput { mods, logical, physical_digit, text, repeat })`; a pure, unit-testable `decide_key(keymap, input, overlay_open) -> KeyDecision { Act, Terminal(bytes), Chrome(EscapeArrowsEnter), Swallow, Ignore }` runs in update where self is available. Decision order preserves every current semantic: keymap hit (respecting pass_through, and Swallowed when an overlay is open and the action is not on the allowlist, which fixes the existing overlay quirk where Ctrl+W typed into the palette closes a pane); else named-key byte table; else `control_byte` passthrough for unmapped Ctrl+letters (unchanged); else composed text. `shortcut_for` is deleted; existing shortcut tests are rewritten against `decide_key`.

**C4. New actions.**
- `FocusProject(n)`: first press selects the Nth workspace in `AppState.workspaces` order (stable sidebar order); repeated presses cycle that project's tabs depth-first with wraparound, reusing the exported `terminal_surfaces` traversal from session_panel.rs so cycle order always matches the visible order. NextTab/PrevTab cycle within the active workspace.
- `GoHome`: dispatches `HomeRequested`; Workstream B's Home view owns the handler (stub until B6 lands).
- `CloseTab` with confirm: child-process detection under the shell is NOT cheaply available (portable-pty exposes only the shell child; tree_kill shells out to taskkill), so the design is confirm-unless-shell-exited behind `terminal.confirmCloseOnRunning` (default true): "running" = `ptys.is_running(id)` locally, live channel for SSH; dead surfaces close immediately. Session-panel X and tab X route through the same gate. Future upgrade path (noted, not v1): a Toolhelp32 child-walk in pandamux-term for true busy detection.
- Clipboard/scroll actions map onto the handlers Workstream A built (CopyOrInterrupt, Copy, Paste, ScrollPageUp/Down).

**C5. Cheat sheet + settings rewiring.** New `Overlay::CheatSheet` (Ctrl+/ or F1, palette item, link in settings) rendering `keymap.sections()` in a two-column categorized modal card following the settings-modal pattern; `kbd_chip` moves to a shared helper. The hand-maintained `default_shortcuts()` list in settings.rs is DELETED; the Settings Keyboard tab renders from the keymap (the existing drift disappears because the render source is the decode source). Workstream C risks: Ctrl+Tab delivery on Windows/winit should be verified on hardware early (fallback rows: Ctrl+PageUp/PageDown); Ctrl+/ needs the F1 fallback on layouts where "/" is shifted; Ctrl+V semantics change from literal 0x16 to paste (pass-through list is the escape hatch; release-note it).

## Cross-workstream contracts (agreed up front)

- **Message names**: `HomeRequested`, `CopyRequested { fallback_interrupt }`, `PasteRequested`, `ScrollPageRequested(i8)`, `ViewportResized/Scrolled/ScrollTo`, `SelectionStarted/Updated/Finished`, `ContextMenuRequested`.
- **Settings keys** (C's `UserSettings` schema is authoritative; B's earlier names fold into it): `terminal.scrollbackLines`, `terminal.welcomePromptEnabled`, `terminal.rightClickPasteOptin`, `terminal.confirmCloseOnRunning`, `keyboard.passThrough`, `keyboard.overrides`, `ui.*`. Close All always confirms (no setting).
- **One confirm overlay**: a single `Overlay::Confirm` unit variant with the payload in runtime state (`pending_confirm: Option<ConfirmAction>` where `ConfirmAction = CloseTab { surface_id } | CloseAll { project_id: Option<ProjectId> }`), used by both C4's close-tab confirm and B5's close-all. One modal component.
- **Shortcut sequencing**: Workstream A lands its clipboard/scroll bindings as additions to the existing `shortcut_for` table (it ships first); Workstream C's keymap refactor then absorbs them as actions (already present in C's default table). No throwaway work: A's handlers are exactly what C's actions dispatch to.
- **Favorites/recents (`LauncherPrefsStore`) stays separate from `UserSettings`**: it is launch data, not preferences, and is written on every launch (recents), so it gets its own file (`config/launcher.json`).
